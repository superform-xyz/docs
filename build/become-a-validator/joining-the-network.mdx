## Overview

Validators are registered by a call to `SuperGovernor.setValidatorConfig()` on Ethereum mainnet. Only the Superform team can call this function during Phase 1. Once your public keys are registered, your node automatically picks up the new configuration via the emitted `ValidatorConfigSet` event â€” no restart required.

## Step 1: Export Your Public Keys

Run `keys export` to get all the values the Superform team needs:

```bash
./bin/oracle-node keys export
```

Output:
```
OCR2 Key Bundle: a1b2c3d4e5f6
  Offchain Public Key (Ed25519):  a60c9e0032...
  Onchain Public Key (ECDSA):     04f62c4a1a...
  Config Public Key (X25519):     8b3f2a1c9d...
  Derived Ethereum Address:       0x742d35Cc...

P2P Key:
  Peer ID: 12D3KooWQzePGqHw66cV1Qsm71eGZKiPEgALYYM3inPtFYibZ67e
  Bootstrap peer entry: 12D3KooWQze...@<host>:<port>
```

## Step 2: Submit Your Key Bundle

Send the following JSON to the Superform team (via your onboarding contact or Discord):

```json
{
  "validator_name": "Your Organization",
  "onchain_public_key": "04abcdef...",
  "ethereum_address": "0x...",
  "offchain_public_key": "abcdef...",
  "config_encryption_public_key": "fedcba...",
  "peer_id": "12D3KooW...",
  "p2p_endpoint": "your-validator.example.com:6690"
}
```

<Warning>
Never include your private keys (`offchain_private_key`, `onchain_private_key`, `p2p_private_key`) in this submission. Only public keys and the peer ID are required.
</Warning>

Key sizes the team will validate:

| Field | Expected Size |
|-------|--------------|
| `onchain_public_key` | 130 hex chars (65 bytes), starts with `04` |
| `offchain_public_key` | 64 hex chars (32 bytes) |
| `config_encryption_public_key` | 64 hex chars (32 bytes) |
| `ethereum_address` | Standard 20-byte Ethereum address |
| `peer_id` | Base58 string, typically starts with `12D3Koo` |

## Step 3: What Happens On-Chain

The Superform team assembles all validator key bundles and calls:

```solidity
SuperGovernor.setValidatorConfig(
    version,             // incremented from the last version
    validators,          // your Ethereum address plus existing validators
    validatorPublicKeys, // your 65-byte ECDSA public key plus existing validators
    quorum,              // updated for the new validator count
    offchainConfig       // ABI-encoded OracleIdentity[] including your P2P and offchain keys
)
```

This emits a `ValidatorConfigSet` event. Every running oracle node subscribes to this event and reconfigures automatically.

## Step 4: Verify Registration

Watch your node logs after the Superform team confirms the transaction. Within a minute of the event being confirmed, you should see:

```
Received ValidatorConfigSet event  blockNumber=... chainID=1
Config loaded successfully  configDigest=0x... signers=N F=...
```

Within the next OCR2 round (`delta_round` â€” 30 minutes on the production v3 config), you will see your node participating in consensus:

```
INFO  Starting PPS observation  epoch=1 round=1
INFO  Observed PPS  vault=0x... pps=1000000000000000000
```

**Verify your config digest matches other validators.** All nodes must report the same `configDigest`. If digests differ, check that your `config_version` in `config.toml` matches the network (see [Configuration Reference](/build/become-a-validator/configuration-reference)).

## Troubleshooting

If the `ValidatorConfigSet` event arrives but config loading fails, the log will contain a specific error:

| Error | Cause |
|-------|-------|
| `validator N (pubkey ...) not found in offchainConfig` | Your onchain public key was not matched in the offchain config â€” contact the Superform team |
| `derived F=N from quorum=N is invalid` | Quorum was not updated correctly after adding your validator â€” contact the Superform team |
| `no ValidatorConfigSet events found` | The transaction has not been submitted yet â€” wait for confirmation |
