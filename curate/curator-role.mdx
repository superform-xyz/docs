---
title: "The Curator Role"
description: "What curators control, what the protocol enforces, and the trust boundaries between them."
---

A curator manages a SuperVault's yield strategy: which protocols receive capital, in what order, under what conditions, and with what risk parameters. The curator does not hold depositor funds. The vault contract does.

This separation is the foundation of the SuperVault security model. Curators have broad operational authority within boundaries the protocol enforces on-chain.

## What Curators Control

| Domain | Capability |
|--------|-----------|
| **Yield source selection** | Whitelist and remove ERC-4626 vaults and Pendle PT positions |
| **Oracle assignment** | Pair each yield source with a pricing oracle |
| **Allocation priority** | Order deposit and redemption ladders to direct keeper behavior |
| **Reserve ratio** | Set the percentage of assets held liquid in the vault |
| **Strategy rules** | Define automated rebalancing, deposit, and withdrawal strategies with the Strategy Engine |
| **Merkle authorization** | Specify exactly which hooks and parameters keepers can execute |
| **Fee configuration** | Set performance and management fees at vault creation |
| **Redeem timelock** | Configure mandatory delay between redemption request and fulfillment |
| **Service controls** | Pause, resume, and adjust frequency of individual keeper workers |
| **Emergency pause** | Halt all vault operations with an on-chain transaction |
| **Access management** | Add secondary managers and view-only users |

## What the Protocol Enforces

Curators operate within constraints set by the SuperGovernor contract and the vault architecture:

| Constraint | Mechanism |
|-----------|-----------|
| **Merkle-verified execution** | Keepers can only call hooks with parameter combinations that match a leaf in the on-chain merkle tree |
| **Timelocked root updates** | New merkle roots must be proposed and wait for a timelock period before they can be executed on-chain |
| **Fee bounds** | Maximum performance and management fee rates are set by SuperGovernor and cannot be exceeded. Performance fees are capped at 50% (5000 bps). |
| **PPS staleness enforcement** | Price-per-share must be updated within a configurable window or vault operations are blocked. The default staleness window is 24 hours and is shown in the dashboard as the Staleness countdown. |
| **$UP upkeep requirement** | Vaults must maintain $UP token balance to cover protocol upkeep costs. On Ethereum, hourly price updates cost roughly 500 $UP per month. On L2s, upkeep costs are negligible. |
| **High-water mark fees** | Performance fees accrue only on gains above the all-time high PPS |
| **Protocol revenue share** | A configurable percentage of fees flows to the Superform protocol. The split is set by governance and displayed in the SuperGovernor configuration at vault creation. |

## Trust Model

SuperVaults operate on a **trust-minimized curator** model:

**Depositors trust the curator to:**
- Select sound yield sources with appropriate risk profiles
- Maintain adequate liquidity reserves for redemptions
- Monitor vault health and respond to adverse conditions
- Keep merkle authorization current as the strategy evolves

**Depositors do NOT need to trust the curator to:**
- Custody their funds (held by the vault contract)
- Execute transactions honestly (merkle-verified on-chain)
- Update PPS accurately (oracle-enforced through off-chain validator computation, EIP-712 signatures, and on-chain consensus checks)
- Respect fee limits (SuperGovernor-enforced)

## Protocol Timelocks

| Change Type | Timelock |
|-------------|----------|
| Material configuration changes (fees, risk limits, asset approvals) | 7 days |
| Global merkle tree hook root updates | 15 minutes |
| Post-unpause skim | 12 hours |

## Dual-Merkle Hook Authorization

Hook execution is gated by two separate merkle roots:

| Root | Controlled By | Contents | Timelock |
|------|---------------|----------|----------|
| **Global hooks root** | Governance | Generic hooks without beneficiary-specific arguments | 15 minutes |
| **Strategy hooks root** | Curator | All global hooks plus beneficiary-specific hooks | 7 days |

A hook must appear in either tree to execute.

## PPS Oracle Model

PPS is not updated by curator discretion alone. Validators compute PPS off-chain, sign EIP-712 typed data, and anyone can submit the update on-chain through `ECDSAPPSOracle.updatePPS()`. The oracle accepts an update only after at least two-thirds of weighted validator stake reaches consensus. Misreporting is backed by slashing and an insurance fund.

## Curator Tiers

| Role | On-chain? | Capabilities |
|------|-----------|-------------|
| **Primary Manager** | Yes | Full vault control. Set at creation, immutable without governance action. |
| **Secondary Manager** | Yes | All operational capabilities except vault settings (redeem timelock) and user management. Added/removed via on-chain transaction. |
| **View Only** | No | Read access to Dashboard and analytics. No write operations. Added/removed off-chain via Erebor API. |

<Warning>
The primary manager key is the root of trust for the vault. If compromised, an attacker could modify yield sources, update merkle authorization, and pause operations. Use hardware wallets and consider multi-sig for production vaults.
</Warning>

## Operational Responsibilities

Running a SuperVault is an active role. Curators are responsible for:

**Continuous monitoring:**
- PPS staleness countdown (vault operations blocked if PPS expires)
- Keeper service health (7 workers must stay operational)
- Redemption queue depth (pending requests need timely fulfillment)
- Yield source performance (APY drift may warrant ladder reordering)

**Periodic maintenance:**
- Merkle tree updates when adding or removing yield sources or hooks
- Ladder rebalancing as market conditions shift
- Strategy rule tuning based on execution history and intent outcomes
- Fee parameter review against competitive benchmarks

**Incident response:**
- Emergency pause if a yield source is compromised
- Service-level pause for targeted worker issues
- Reserve adjustment during high redemption demand
- Post-mortem analysis via execution history and audit logs
