## config.toml

Copy `config.template.toml` to `config.toml` and fill in your values. The file is structured into sections described below.

### `[node]`

```toml
[node]
node_id = "oracle-1"        # Unique identifier for this node (used in logs)
p2p_listen_addr = "0.0.0.0" # Address to listen on for P2P connections
p2p_port = 6690              # TCP port for ragep2p
```

### `[keys]`

Choose one of three provisioning methods.

**Option 1 — Direct hex (recommended for production)**

```toml
[keys]
offchain_private_key = "76918f97bfe8..."  # Ed25519 private key, 128 hex chars

[keys.onchain_private_key]
# Use one of:
private_key = "ed6ca3d3ac17..."           # ECDSA secp256k1, 64 hex chars (dev/testing)
# kms_key_id = "arn:aws:kms:..."          # AWS KMS key ID (production)
```

**Option 2 — Key file (development only)**

```toml
[keys]
key_file_path = "./keystore/ocr2_<id>.json"
```

**Option 3 — AWS KMS for onchain key**

```toml
[keys.onchain_private_key]
kms_key_id = "arn:aws:kms:us-east-1:123456789:key/abcd-1234"
```

Supported KMS key ID formats: full ARN, alias (`alias/name`), or UUID.

### `[p2p]`

```toml
[p2p]
bootstrap_peers = [
    "12D3KooWGc9hJKBWks3oq9e7H4oZseoncbrxcZQfkW5NTBH473un@bootstrap-1.supervaults.superform.xyz:6690",
    "12D3KooWCDXcCoE86B4dfMg5hXpoPr6xj4wGRaYqxHbkgPZW3yer@bootstrap-2.supervaults.superform.xyz:6690",
]
# announce_addresses = ["203.0.113.42:6690"]  # Required if behind NAT
```

`announce_addresses` tells other validators what address to dial when connecting to you. Set this if your public IP differs from the address the node listens on (e.g., behind a load balancer or NAT gateway).

### `[database]`

```toml
[database]
url = "postgresql://oracle:password@localhost:5432/oracle_db?sslmode=disable"
max_connections = 10
max_idle_conns = 5
conn_max_lifetime = 3600  # seconds
```

### `[monitoring]`

```toml
[monitoring]
metrics_port = 9090         # Prometheus /metrics scrape target
health_port = 8080          # /healthz and /readyz endpoints
log_level = "info"          # debug | info | warn | error
log_format = "json"         # json | console

# Warn (via Prometheus gauge + logs) if no PPSValidated on-chain event
# arrives within this window. Set lower for high-frequency deployments.
health_check_interval = "1h"

# Reject reports where the leader's PPS deviates more than this from
# your own observation. 0 = disabled. 10_000_000 = 1% tolerance.
max_pps_deviation_ppb = 10000000
```

### `[ocr2]`

```toml
[ocr2]
# Selects config/v{N}.json for OCR2 timing parameters.
# All nodes MUST use the same version — mismatched versions produce
# different config digests and prevent consensus.
config_version = 3

# Chain ID where SuperGovernor is deployed (default: 1 = Ethereum mainnet)
# config_sync_chain_id = 1
```

### `chain_config_path`

```toml
chain_config_path = "./chains.yaml"
```

Points to a YAML file that defines each chain the node observes. See the [`chains.yaml` reference](#chainsyaml) below.

## chains.yaml

`chains.yaml` is a map of chain ID to chain configuration. Copy `chains.template.yaml` and fill in your RPC endpoints and the addresses provided by the Superform team during onboarding.

```yaml
1:                          # chain ID (top-level key)
  name: ethereum            # human-readable label (used in logs and metrics)
  chain_id: 1               # must match the top-level key
  https_rpc: "https://eth-mainnet.g.alchemy.com/v2/YOUR_KEY"
  wss_rpc:   "wss://eth-mainnet.g.alchemy.com/v2/YOUR_KEY"
  block_time: 12s           # expected block time (Go duration format)
  supported_strategies:     # vault contract addresses to observe on this chain
    - "0xVault1Address"
    - "0xVault2Address"
  ecdsa_oracle_address:   "0x..."  # ECDSAPPSOracle contract — provided by Superform
  super_governor_address: "0x..."  # SuperGovernor contract — provided by Superform
```

### Field reference

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Human-readable chain label. Used in logs and Prometheus `chain_id` labels. |
| `chain_id` | integer | EVM chain ID. Must match the top-level map key. |
| `https_rpc` | string | HTTPS RPC endpoint. Used for contract reads and transaction submission. |
| `wss_rpc` | string | WebSocket RPC endpoint. Used for subscribing to `ValidatorConfigSet` and `PPSValidated` events. |
| `block_time` | duration | Expected time between blocks (Go duration: `12s`, `2s`, `1s`). Used to calibrate event polling. |
| `supported_strategies` | string[] | Vault contract addresses whose `pricePerShare()` your node will call each round. Provided by Superform during onboarding. |
| `ecdsa_oracle_address` | string | Address of the `ECDSAPPSOracle` contract where PPS reports are submitted. Provided by Superform. |
| `super_governor_address` | string | Address of the `SuperGovernor` contract that emits `ValidatorConfigSet` events. Provided by Superform. Only needs to be set on the config sync chain (Ethereum mainnet, chain ID 1). |

### Supported chains

The Superform team will tell you which chains to include and provide the contract addresses for each during onboarding. The template includes Ethereum (1), Base (8453), Arbitrum (42161), Optimism (10), and Polygon (137).

Only include chains where you have a reliable RPC endpoint. A failing RPC on any configured chain causes observation errors for all vaults on that chain, which counts against your participation rate.

### RPC provider requirements

- **HTTPS** — required for all chains
- **WebSocket (WSS)** — required for event subscriptions; the node will warn on startup if WSS is unavailable
- Both endpoints should be from a reliable provider (Alchemy, QuikNode, or a self-hosted node). Public rate-limited endpoints are not suitable for production validators.

## OCR2 Timing Configurations

The `config/v{N}.json` files control OCR2 protocol timing. **All nodes must use the same version** — a mismatch produces different config digests and the node cannot participate in consensus.

### v2 — High-Frequency (Testing / Sub-Minute Updates)

2-second rounds. Use for testnets or when vault strategies require PPS updates more frequently than once per hour.

| Parameter | Value | Meaning |
|-----------|-------|---------|
| `delta_round` | 2s | Round start interval |
| `delta_progress` | 120s | Leader stuck timeout |
| `delta_resend` | 30s | Unconfirmed message resend interval |
| `delta_grace` | 5s | Grace period for late observations |
| `delta_stage` | 10s | Time per protocol stage |
| `r_max` | 3 | Rounds per epoch (6s epoch) |
| `max_stale_duration` | 5m | Maximum report age before rejection |

### v3 — Hourly Production (Default)

30-minute rounds. Use for production SuperVaults with a ~1-hour minimum PPS update interval and 24-hour maximum staleness.

| Parameter | Value | Meaning |
|-----------|-------|---------|
| `delta_round` | 30m | Round start interval |
| `delta_progress` | 10m | Leader stuck timeout |
| `delta_resend` | 5m | Unconfirmed message resend interval |
| `delta_grace` | 1m | Grace period for slow multi-chain RPCs |
| `delta_stage` | 2m | Time per protocol stage |
| `r_max` | 5 | Rounds per epoch (2.5h epoch) |
| `max_stale_duration` | 5m | Maximum report age before rejection |

### How Staleness Works

The system enforces freshness at four levels:

| Tier | Config Location | v3 Default | Purpose |
|------|----------------|------------|---------|
| Attempt frequency | `delta_round` in `v{N}.json` | 30m | How often the protocol attempts a round |
| Report freshness | `max_stale_duration` in `v{N}.json` | 5m | Rejects reports that took too long to finalize |
| On-chain monitoring | `health_check_interval` in `config.toml` | 1h | Alerts when no on-chain confirmation arrives |
| Contract enforcement | `maxStaleness` on SuperVaultAggregator | 24h | Auto-pauses the strategy if exceeded |

### Switching Versions

Update `config_version` in `config.toml` on all nodes and restart them simultaneously. Because all nodes derive the config digest from the same version file, mismatched restarts cause a split and stall consensus until all nodes are on the same version.
